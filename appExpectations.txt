LATEST BUILD ROUND — see BACKLOG.md for the full deduped checklist. Newly shipped:
✅ Scheduling permission lockdown: Scheduling + Visits are visible ONLY to
   scheduler-and-up (Owner / Admin / Clinical Director / Nurse Supervisor /
   Scheduler). Field staff (RN/LPN/HHA/CNA/Caregiver) no longer have
   scheduling:read — they cannot see the agency schedule, the roster, or assign
   caregivers. They see ONLY their own assigned shifts (My Shifts) and a personal
   caregiver dashboard. EVV check-in/out moved to My Shifts with own-visit
   ownership enforced server-side; the agency-wide EVV board is a supervisor monitor.
✅ Patient requests: request a SPECIFIC caregiver (chosen from caregivers who have
   worked with them) and request an ADDITIONAL visit; scheduler approval reassigns
   or creates the visit (scope-of-practice verified).
✅ Messaging rule enforced: conversation participants model; a caregiver may only
   message a supervisor/scheduler; threads are participant-scoped (admins oversee all).
✅ Self-serve SIGNUP: creates a new agency and makes the registrant its Agency Owner
   (role fixed server-side; email verification). Foundation for SaaS multi-agency.
✅ Discipline-specific demo logins: rn@ / lpn@ / hha@ / cna@ (+ generic caregiver@),
   each linked to a caregiver record with upcoming assigned shifts.
✅ WHITE LABEL — runtime multi-agency (per the white-label workflow spec):
   - Tenant resolved from the HOST at request time: custom domain (agency_domains)
     → platform subdomain (Agency.slug) → logged-in user's agency → platform default.
   - Branding is DATA not code: portal name, logo, favicon, login banner, support
     email/phone, PDF footer; the WHOLE brand palette re-themes from one primary
     color (Tailwind brand scale now CSS-variable channels overridden by <BrandStyle>).
   - Admin panel: branding fields + custom-domains manager (/api/admin/domains) +
     platform subdomain slug. Login/signup pages brand by host; PDFs carry agency footer.
   - Second demo agency seeded — "Sunrise Care" (blue) at http://sunrise.localhost:3000
     vs Cura_Sera (teal) at http://localhost:3000 — same code, different data, isolated.

Security must happen at:

- Database level
- Backend API level
- Role permission level
- Row-level filtering
- Audit logging level

Every request checks:

1. Is user logged in?
2. What agency?
3. What branch?
4. What role?
5. What permission?
6. Is the patient assigned?
7. Is the shift approved?
8. Is medical access allowed?
9. Should this action be logged?

If not allowed:

Access denied.
